# 发布收尾清单

> 规则：每完成一项，就在这里勾掉一项。

## 第一阶段：发布路线收敛

- [x] 明确仓库按应用交付，不按 npm 包发布
- [x] 在 README 中写明正式交付方式：源码启动 / Docker / 可选 GitHub Release
- [x] 废弃不适用的 npm publish 任务

## 第二阶段：仓库合法与安全前置

- [x] 增加 `LICENSE`（MIT）
- [x] 在 `package.json` 中补充 `license: "MIT"`
- [x] 检查仓库中是否存在明显 token / credential 残留
- [x] 检查 GitHub 仓库是否已配置 `NPM_TOKEN`

## 第三阶段：应用发布路径整理

- [x] 明确 GitHub Release 是否作为正式发布入口
- [x] 如果启用 GitHub Release，补充版本号、发布说明和发布流程约定
- [x] 明确 Docker 发布策略（本地 build、镜像标签、是否推送镜像仓库）
- [x] 在 README 或单独文档中补充应用升级与回滚流程

## 第四阶段：可选后续

- [x] 如有需要，增加 GitHub Release 操作文档
- [x] 如有需要，增加 Docker 镜像发布或部署自动化

## 当前决议

- **源码检出 + 本地启动**：主交付方式，面向个人使用。
- **Docker / Compose**：正式支持的自托管方式，默认从仓库源码目录本地构建，不要求预先发布公共镜像。
- **GitHub Release**：作为正式版本记录入口，用于发布版本号、变更说明、升级提示与源码快照；不是二进制安装包分发渠道。
- **镜像仓库推送**：当前不作为必需项，只有在明确需要跨机器复用预构建镜像时再进入范围。
- **npm 包发布**：不适用于该仓库，保持排除。
